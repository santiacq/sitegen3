nota

github · releases

A command-line note-taking tool that stores everything in a single SQLite database. Notes are written in Markdown, full-text search is built in, and the entire state lives in one portable file you can back up with cp.

motivation

I used to keep notes in plain files. That worked until I had enough of them that finding things became slow and unreliable. I tried a few dedicated apps — Notion, Obsidian, Logseq — and bounced off all of them for the same reason: they put their structure on top of my notes.

nota has no structure beyond a flat list of entries. It gets out of the way.

features

  • Write notes in Markdown — rendered in the terminal with color
  • Full-text search across all notes using SQLite FTS5
  • Tag notes and filter by tag
  • Open any note in $EDITOR for longer edits
  • Export to HTML, plain text, or JSON
  • Single binary, no config required

quick start

installation

Download the binary for your platform from the releases page, or build from source:

git clone https://github.com/you/nota
cd nota
go build -o nota ./cmd/nota
sudo mv nota /usr/local/bin/

first steps

  1. Add a note: nota add "first thought"
  2. Add a longer note in your editor: nota edit
  3. Search: nota search "first"
  4. List recent notes: nota ls

By default, nota stores the database at ~/.local/share/nota/notes.db. Set the NOTA_DB environment variable to use a different path.

usage

adding notes

# Quick one-liner
nota add "meeting with team at 3pm"

# With tags
nota add --tag work --tag meeting "sync with design team"

# Open editor for a longer note
nota edit

searching

# Full-text search
nota search "sqlite performance"

# Filter by tag
nota ls --tag work

# Search within a tag
nota search --tag work "meeting"

example output

$ nota search "sqlite"
#42  2026-03-14  [work, research]
SQLite is surprisingly capable for this use case. FTS5 is fast enough
that I don't need a dedicated search index for my notes collection.

#38  2026-02-28  [dev]
Looked into sqlite-vec for vector search. The API is clean but the
serialization format for float32 arrays is not obvious from the docs.

configuration

nota reads a config file at ~/.config/nota/config.toml if it exists. All fields are optional.

Key Type Default Description
db string ~/.local/share/nota/notes.db Path to the SQLite database
editor string $EDITOR Editor to open for nota edit
pager string less -R Pager for long output
color bool true Enable terminal color output
date_format string 2006-01-02 Go time format for displaying dates

example config

# ~/.config/nota/config.toml
db     = "~/Dropbox/nota.db"
editor = "nvim"
color  = true

design decisions

why SQLite

The entire state is one file. Backups are trivial. Corruption is recoverable. FTS5 is fast and built in. There is no server to run or daemon to keep alive. SQLite is boring in the best possible way.

why not a file-per-note approach

Files are fine for small collections but fall apart for search. The options are:

  • grep — fast, exact match only; misses semantically related notes
  • Ripgrep — faster grep, same limitation
  • External indexer (Meilisearch, Typesense) — good search, but now you have a server to run
  • SQLite FTS5 — full-text search built into the same file as the data; no moving parts

Every tool you don't have to run is one less thing that can break at 11pm when you just want to find a note.

status

Stable. I've been using nota daily for about a year. The database has ~3,000 entries and search is still instant.

The core feature set is complete. Future work, if any, will be small quality-of-life improvements:

  • Better terminal rendering for tables in Markdown
  • An import command for bulk-loading plain text files

PRs welcome for bugs. Open an issue before starting on a new feature — the goal of this project is to stay small.
