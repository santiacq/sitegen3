Building a personal search engine

I've been keeping notes for about ten years. Plain text files, scattered across machines, synced imperfectly. At some point the collection became large enough that I couldn't find things anymore. So I built a search engine for it — and learned more than I expected about how search actually works.

The problem with grep

The obvious first tool is grep. It works until it doesn't. The failure mode is always the same: you know you wrote something down, you can feel the shape of the thought, but you can't remember the exact words you used. grep requires you to already know what you're looking for.

A search engine is useful precisely when your memory has failed. If you remembered the words, you wouldn't need to search.

Full-text search solves the exact-match problem. But semantic search — matching by meaning rather than string — is what I actually needed. That means embeddings.

How embeddings work

An embedding model maps a chunk of text to a point in high-dimensional space. Texts with similar meaning end up close together. The key insight is that you can embed both your documents and your query into the same space, then find the nearest neighbors.

Chunking strategy

Before embedding anything, you need to decide how to split your documents. The options, roughly in order of sophistication:

  1. Fixed-size chunks — split every N characters. Simple, but cuts across sentence boundaries.
  2. Sentence-based chunks — split on punctuation. Better for prose, breaks down for code.
  3. Paragraph-based chunks — split on blank lines. Works well for notes; this is what I use.
  4. Semantic chunking — use a model to find natural topic boundaries. High quality, high cost.

I went with paragraph-based chunking. Notes are already written in paragraphs; it felt right to respect that structure.

Choosing a model

I compared three embedding models on a sample of my notes:

Model Dimensions Speed Quality
all-MiniLM-L6-v2 384 fast good
nomic-embed-text 768 medium very good
text-embedding-3-large 3072 API call best

I ruled out text-embedding-3-large immediately — I want this to run offline. nomic-embed-text runs locally via Ollama and the quality difference over MiniLM was noticeable enough to justify the slower indexing time.

The index

Once you have embeddings, you need a way to query them. The brute-force approach — compute cosine similarity against every vector on every query — is fine up to about 100k chunks. Beyond that, you want an approximate nearest-neighbor index.

I tried two options:

  • FAISS — Facebook's library, very fast, C++ under the hood, slightly awkward Python bindings
  • SQLite + sqlite-vec — a vector extension for SQLite; slower than FAISS but the whole index is a single file I can copy around

I chose SQLite. My notes collection is around 40k chunks — well within the range where the performance difference doesn't matter, and having a portable single-file index is worth a lot.

Schema

CREATE TABLE chunks (
  id      INTEGER PRIMARY KEY,
  path    TEXT NOT NULL,
  mtime   INTEGER NOT NULL,
  content TEXT NOT NULL
);

CREATE VIRTUAL TABLE chunk_vss USING vec0(
  embedding FLOAT[768]
);

Indexing a file

def index_file(path: str, conn: sqlite3.Connection) -> int:
    mtime = os.path.getmtime(path)
    text = open(path).read()
    paragraphs = [p.strip() for p in text.split("\n\n") if p.strip()]

    indexed = 0
    for para in paragraphs:
        embedding = embed(para)  # returns list[float]
        conn.execute(
            "INSERT INTO chunks (path, mtime, content) VALUES (?, ?, ?)",
            (path, mtime, para),
        )
        chunk_id = conn.lastrowid
        conn.execute(
            "INSERT INTO chunk_vss (rowid, embedding) VALUES (?, ?)",
            (chunk_id, serialize_float32(embedding)),
        )
        indexed += 1

    return indexed

Querying

The query path is straightforward: embed the query string, find the k nearest vectors, return the corresponding chunks with their source files. 

def search(query: str, k: int = 10) -> list[dict]:
    q_embedding = embed(query)
    rows = conn.execute("""
        SELECT c.path, c.content, v.distance
        FROM chunk_vss v
        JOIN chunks c ON c.id = v.rowid
        WHERE v.embedding MATCH ?
          AND k = ?
        ORDER BY v.distance
    """, (serialize_float32(q_embedding), k)).fetchall()
    return [{"path": r[0], "content": r[1], "score": r[2]} for r in rows]

A note on scoring

The distance returned by sqlite-vec is L2 distance in the embedding space, not cosine similarity. For normalized vectors these are equivalent, but nomic-embed-text does not normalize by default. I added an explicit normalization step after embedding: 

import numpy as np

def embed(text: str) -> list[float]:
    raw = model.encode(text)
    norm = raw / np.linalg.norm(raw)
    return norm.tolist()

The interface

I considered building a web UI but didn't want to run a server. The interface is a small terminal program: type a query, get back a ranked list of excerpts with file paths. Press Enter on a result to open the file at the relevant line in $EDITOR.

The right interface for a tool is the one you'll actually use. I live in the terminal, so the terminal is right.

What I learned

A few things surprised me building this:

  • Chunking matters more than the model. Bad chunking with a good model produces worse results than good chunking with a mediocre model.
  • Incremental indexing is worth building from the start. Re-indexing 40k chunks takes several minutes; skipping unchanged files (via mtime) brings that to seconds.
  • The hardest part wasn't the search — it was deciding what to put in the index. I ended up excluding meeting notes and todo lists entirely.
